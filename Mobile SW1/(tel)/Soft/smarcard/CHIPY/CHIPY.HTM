What is "CHIPY"?

"CHIPY" is a (powerfull) reader/analyser program.

It may be downloaded at the end of this page.

What is "CHIPY" able to do?

"CHIPY" is a reader/analyser program able to:

Read a telecard from the parallel port of a PC (see "What you need to know about electronics telecards" for a schema of the reader)
Display a dump of the card's memory (both binary and hexadecimal)
Analyse the card's memory and display information about the card using an easily modifiable definition file
Save dump and analyse result to a file (using MSDOS's >> redirection)
Read a dump previously saved to a file (either generated by "CHIPY" or by another program)

What can "CHIPY" analyse in a card?

"CHIPY" uses an easily modifiable definition file (CHIPYDEF.TXT) to analyse the cards. The informations "CHIPY" will display for a card depends on what is defined for this card. Depending on your definition "CHIPY" will be able to display:

Country of emission
Usage of the card (telecard, parking card, car wash...)
Issuer of the card (name of the operator...)
Facial value of the card (in units)
Used/remaining value of the card (in units)
Convert units in money if the value of each unit is known
Display an area of the card with simple calculations if various formats (for exemple serial number...)
Display some user defined text conditionnaly

So you can share dumps, and .DEF file definitions to figure out the mapping of telecards...

How to make a CHIPYDEF.TXT file?

A CHIPYDEF.TXT file is already available on this page, some cards listed in "What you need to know about electronics telecards" are defined in this file (all cards will be defined soon).

You may add your own definitions (and share them so I add them in the next version of CHIPYDEF.TXT, thanks).

A .DEF file is made of several cards definitions, each beginning by a 'CARD' line, each card definition is made of multiple lines:

CARD      : Introduce a card definition
VERIFY-OR : Check the value of mutiple areas of the card (EQual, Not Equal) to distinguish cards
CHECKSUM  : Some cards have one (or multiple) checksum to check if the header of the card is not altered
VALUE     : Define the facial value (in units) of the card, different ways to define the facial value are supported
CURRENCY  : Used to define the cash value of one unit (currency symbol, significant digits)
CREDIT    : Define the way used or remaining units are stored in the card
DISPLAY   : Used to display the contents of a part of a card with simple calculations (ie. for serial numbers)
TEXT      : Used to display a text (always, or under simple condition)

"CHIPY" works by reading the card's memory (from parallel port or from a dump stored in a file), reading the cards definitions from the CHIPYDEF.TXT file, and then for each definition checks (in this order): VERIFY, CHECKSUM, CREDIT.

INTRODUCTION

Bits ('b') and Bytes ('B') are numbered starting from zero.

b(0..7) is the area starting at bit number 0 and ending at bit number 7 (it's lenght is 8 bits). b(7..0) is the area beginning at bit number 7 and starting at bit number 0, so the area is read in reverse). All

Notations in use in this document:

[parameter] indicates an optionnal parameter
CAPS indicates a litteral
| indicates OR
$ indicates an hexadecimal value

To undestand the exemples please refer to "What you need to know about electronics telecards"

CARD: Introduce a card definition

CHIPYDEF.TXT file is made of several cards defintions, 'CARD' is used to introduce each definition.

CARD "name",["country"],["usage"],["issuer"]

  • name: a name for the card definition, this name will be displayed when a card definition is matched.
  • country: the name of the country where the card is issued
  • usage: the usage of the card (telecard, parking card, car wash...)
  • issuer: the name of the issuer (operator...)

Only "name" is required.

Exemple: CARD "T1G","France","Telecard","France Telecom"

VERIFY-OR: Distinguish cards

Some areas of the cards are used as a 'marker' of a type of card. VERIFY is used to checks these areas.

VERIFY start,end,EQ|NE,value
 
Two consecutives VERIFY are a logical 'AND'.

Exemple: VERIFY 8,15,EQ,$83
         VERIFY 80,95,EQ,$115E
May be used to verify if a card is a telecard from Mali.

OR may be used after a first VERIFY to make a logical 'OR'.

Exemple: VERIFY 8,11,EQ,$0

         OR     8,15,EQ,$80

         VERIFY 80,87,EQ,$10

         OR     80,87,EQ,$14

May be used to verify if a card is a first generation telecard from France Telecom.

CHECKSUM: Compute Checksum

Mainly on first generation telecard, CHECKSUM are used to check if the header area is not altered. Checksum is stored in an area of the card and is computed against another area of the card, the result of the computation must be equal to the first area. Some cards have up to 3 checksums. If mutliple checksums are defined, all of them must match.

CHECKSUM startchecksum,endchecksum,startzone,endzone,seed,value

Exemple: CHECKSUM 0,7,8,31,$E3,-4
         CHECKSUM 32,39,40,63,$E3,-4
         CHECKSUM 64,71,72,95,$E3,-4
May be used to verify the checksums of a first generation telecard from France Telecom.

VALUE: Compute facial value of a card

The facial value (in units) of a card may be stored somewhere in the header area. On first generation and french second generation telecards this is mandatory because these cards counts the USED units, so the application (public phone...)  needs to know the facial value of the card. Eurochip (Germany, Greece, Netherlands...) counts the REMAINING units, so there is no need to know the facial value of the card (for these cards facial value may be coded in the card or not).

VALUE represents the facial value of the card, the number of units you can burn by using the card. Some manufacturers burn units in factory to test the card, these units are not in the card's units, the parameter BurnedUnits is used for it.

VALUE can be defined by a TABLE, by calculation from a BINARY zone of card or a BCD zone of a card.

VALUE TABLE,Start,End
VALUE String,FacialValue,BurnedUnits
 " "   "  "   " "   "       "
VALUE String,FacialValue,BurnedUnits

Exemple: VALUE TABLE,88,95
         VALUE $13,120,10
         VALUE $07,60,10
         VALUE $06,50,10
         VALUE $05,40,10
         VALUE $04,25,15
         VALUE $02,5,15
May be used to define the value of a first generation telecard from France Telecom.

For BINARY and BCD the number stored in the area from in area (Start:End) is multiplied by Multiplier. Because on some cards the units burned in factory are included in the VALUE computed and sometimes it is not included you need to do some special maths:

  • If BurnedUnits is positive, result of the calculation is the facial value of the card number, the number of burned units is to be added from the result of calculation to obtain the physical number of units of the card.
  • If BurnedUnits is negative, result of the calculation is the number of physical number of units of the card, the number of burned is to be substracted from the result of the calculation to obtain the facial value.

VALUE BINARY Start,End,Multiplier,BurnedUnits
VALUE BCD    Start,End,Multiplier,BurnedUnits

Exemple: VALUE BINARY,88,95
May be used to define the value of card from Mexico (Ladatel).

         VALUE BCD,20,31,1,-2
May be used to define the value of card from Mali (Sotelma).

CURRENCY: Define the cash value of a unit

CHIPY does all his calculations in Units, when the facial value of a card is given in cash then you can use the CURRENCY parametrer to convert Units in Money.

WARNING: You can not use CURRENCY in all coutries, for example french telecards 50 units are sold 40.60 FF V.A.T. included, when the V.A.T. changes the price of telecards changes, in Martinique (french district) V.A.T. is lower than in Paris, so the price of telecards is not the same. You can use CURRENCY when the price of the card is printed on the card itself and is not subject to change.

CURRENCY UnitValue,NumberOfDecimals,CurrencySymbol
Exemple: CURRENCY 0.1,2,"FF"
May be used to define the value of a Paris Carte first generation (each unit cost 0.10 FF, prices are written with two decimal digits, "FF" stands for French Francs.

CREDIT: Define the way used or remaining units are stored in the card

Remaining or used credit can be stoted in many ways. First of all, there are two families of cards:

  • Cards which count the USED units (mainly first generation telecards, french second generation telecards), so each unit has a negative value (there is ALWAYS a way to know the facial value of these cards)
  • Cards which count the REMAINING units (mainly Eurochip: Germany, Greece, Netherlands...), so each unit has a positive value (facial value of the card is not always known for theses cards).

You may code several CREDIT lines for a single card because there may be multiple areas in the card used to store the CREDIT (each area have a different unit value), for example Eurochip and french T2G use several CREDIT lines, one for each stage of the octal counter). You can not mix positive and negative CREDIT lines in a single card.

The first way to store the CREDIT is the 'Unit Area', it is used on first generation telecards (and for each stage of an octal counter).

The area is initially set to '0', each time a unit is used a '0' is set to '1' starting by the first bit of the area. You can not have a '0' in the middle of a serie of '1'.

0000 0000  0000 0000             0 Units   (remaining or used
1111 0000  0000 0000             4 Units    depending of the
1111 1000  0000 0000             5 Units    card's family)

1111 1010  0000 0000             Invalid (because of the 0)
Examples of 'Unit Areas'

CREDIT UNIT,Start,End,BitValue

Exemple: CREDIT UNIT,96,247,-1
May be used to define the credit of a first generation telecard from France Telecom.

French T2G and Eurochip use octal counters to store the CREDIT of the card. Octal counter is a way to increase the number of units stored in a card, because with a single Unit area number of units is limited by the memory size of the card (each bit is a unit).

Octal counter is a powerfull mecanism, CREDIT is stored using several unit area (called stage), each of them represents a digit of an octal number (right digit has a weight of 1, next 8, next 64 and so on...).

The Unit Area of each stage is read from right to left, so Start is greater than End on the CREDIT line.

c(512)   c(64)    c(8)     c(1)         Weight
64    71 72    79 80    87 88     95    Address
|      | |      | |       | |      |
00000000 00000000 00000000 00000000    0*512 + 0*64 + 0*8 + 0*1 = 0 Units
00000000 00000000 00000001 00000001    0*512 + 0*64 + 1*8 + 1*1 = 9
00000000 00000000 00000111 00011111    0*512 + 0*64 + 3*8 + 5*1 = 29
00000000 00000001 00011111 00000111    0*512 + 1*64 + 5*8 + 3*1 = 107

00000000 00000101 00011111 00001111    c(64) is Invalid
Examples of 'Octal counters'

Exemple: CREDIT UNIT,71,64,-512
         CREDIT UNIT,79,72,-64
         CREDIT UNIT,87,80,-8
         CREDIT UNIT,95,88,-1
May be used to define the credit of a second generation telecard from France Telecom (T2G).

Eurochip (Germany, Greece, Netherlands...) also uses an Octal counter to store the CREDIT of a card, Eurochip counts the REMAINING units of card, so BitValue is positive.

Exemple: CREDIT UNIT,71,64,4096
         CREDIT UNIT,79,72,512
         CREDIT UNIT,87,80,64
         CREDIT UNIT,95,88,8
         CREDIT UNIT,103,96,1
May be used to define the credit of a German card.

A 'Group area' is another way to store the credit of a card,it is similar to Unit Area, but, when several units a are burned at the same time only the last one is set from '0' to '1'. Spain uses 2 Group Areas to store the credit of the card.

0000 0000  0000 0000  0000 0000     0  Units
0100 0000  0000 0000  0000 0000     2   Units
0100 0001  0000 0000  0000 0000     8   Units
0100 0001  0000 1000  0000 0000     13 Units
Examples of 'Group Areas'

CREDIT GROUP,Start,End,BitValue

Exemple: CREDIT GROUP,205,106,-1
         CREDIT GROUP,255,206,-2
May be used to define the credit of a 1000pta spanish card.

On some cards (for example Paris Carte second generation) units are stored in a 'Count Area', you count the bits set to '1' in the area to read the credit of the card.

0000 0000  0000 0000  0000 0000     0  Units
0001 0000  0000 0000  0000 0000     1   Unit
0001 0010  0000 0000  0000 0000     2   Units
1011 0010  1100 0100  0000 0000     7 Units
Examples of 'Count Areas'

CREDIT COUNT,Start,End,BitValue

You can mix CREDIT types in a single card (for example Paris Carte second generation uses a Unit area and a Count Area).

Exemple: CREDIT UNIT,319,114,1
         CREDIT COUNT,352,415,206
May be used to define the credit of a Paris Carte second generation.

Some other cards work with 'Token Area' (for example 'Paris Carte First Generation'). These cards use multiple Unit Areas (each bit is a 'Token'), the value of each Token in the Unit Area depends on the Token Type.

 16  0128: 1110 1110  0000 0000  0000 0000  0000 0000

 20  0160: 1011 1100  0000 0000  0000 0000  0000 0000

 24  0192: 1001 1100  0000 0000  0000 0000  0000 0000

 28  0224: 1101 1110  0000 0000  0000 0000  0000 0000

 32  0256: 1100 1000  0000 0000  0000 0000  0000 0000

 .....

 156 1248: 0000 0000  0000 0000  0000 0000  0000 0000

           ^    ^

           |    |

           |    Tokens

           Token Type
Example of Token Area

CREDIT TOKEN,TokTypeStart,TokTypeEnd,TokZoneStart,TokZoneEnd,Offset,NbZones
CREDIT TOKEN,TokenType,Value   (as many lines as needed)

TokTypeStart and TokTypeEnd are the start and end address of the first Token Type (in this case 128,131). TokZoneStart and TokZoneEnd are the start and end address of the first Token Zone (in this case 132,159). Offset is the offset between two consecutives Token Zones (in this case 32 bits), NbZones is the number of Token Zones in the card (in this case 36).

Exemple: CREDIT TOKEN,128,131,132,159,32,36
         CREDIT $E,-200
         CREDIT $D,-100
         CREDIT $C,-50
         CREDIT $B,-20
         CREDIT $A,-10
         CREDIT $9,-5
         CREDIT $8,-2
         CREDIT $7,-1
May be used to define the credit of a Paris Carte first generation

NOTE: CREDIT TOKEN can not be mixed with other CREDIT types.

Two other types of CREDIT exist, there are used when the credit is stored in binary or BCD format.

CREDIT BINARY Start,End,Multiplier
CREDIT BCD    Start,End,Multiplier
